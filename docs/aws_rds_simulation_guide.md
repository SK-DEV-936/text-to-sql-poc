# AWS RDS MySQL Simulation Guide

The Boons Text-to-SQL agent is natively designed to swap between local execution and AWS services without any required code changes. By default, it uses a high-fidelity MySQL Docker container for local development to simulate AWS RDS.

## 1. Environment Switching
The application uses the `ENVIRONMENT` variable (in `.env`) to toggle between scenarios:

- `ENVIRONMENT=local`: Uses the local Docker MySQL container (`localhost:3306`).
- `ENVIRONMENT=aws-dev`: Uses the AWS RDS Dev instance.
- `ENVIRONMENT=aws-prod`: Uses the AWS RDS Production instance.

## 2. Configure Environment Variables
Create or open the `.env` file in the root of your project directory (`boons-text-to-sql-agent/.env`). 

```env
# Toggle the environment
ENVIRONMENT="local"

# Database Settings (Only override if needed, defaults are in config.py)
# For local, these point to your Docker container:
DB_HOST="localhost"
DB_PORT=3306
DB_USER="boons_readonly"
DB_PASSWORD="change-me"
DB_NAME="boons"

# AWS Bedrock/OpenAI Settings
# OPENAI_API_KEY="sk-..."
```

## 3. Start the Application
Instead of using the `./run_demo.sh` wrapper (which hardcodes the local executor), activate your virtual environment and start the FastAPI uvicorn daemon manually using your updated `.env` settings:

```bash
# 1. Activate your python environment
source venv/bin/activate

# 2. Start the FastAPI backend
uvicorn boons_text_to_sql_agent.main:app --port 8000
```

*In a separate terminal tab, you can start the Streamlit UI to connect to it:*
```bash
source venv/bin/activate
streamlit run demo_chat.py
```

## 4. RAG & Knowledge Retrieval (Bedrock vs Local)
To ensure the LLM understands the schema, business logic, and semantic synonyms, the application utilizes Retrieval-Augmented Generation (RAG).

- **Local Execution**: When `ENVIRONMENT=local`, the application builds and queries a local `FAISS` vector store in-memory using OpenAI Embeddings.
- **AWS Execution**: When `ENVIRONMENT=aws-prod` or `aws-dev`, the application utilizes **Amazon Bedrock Knowledge Bases**. 
  - *Infrastructure*: AWS automatically provisions an **Amazon OpenSearch Serverless** vector index under the hood. 
  - *Ingestion*: You simply upload the database documentation/schema json to an S3 bucket, and Bedrock automatically chunks, embeds (using Titan Embeddings), and syncs it to OpenSearch.
  - *Retrieval*: The LangChain `AmazonKnowledgeBasesRetriever` seamlessly handles querying this AWS-native index.

## 5. How It Works Under The Hood
When `ENVIRONMENT` is set to any non-local value, the `main.py` routing logic instantiates the `MySqlExecutor` class (located in `infrastructure/db/mysql_executor.py`). 

This class utilizes the asynchronous `aiomysql` driver to open non-blocking connections to your AWS RDS instance. Because the LLM strictly generates MySQL dialect (`LangChainTextToSqlAdapter`) and retrieves context via Bedrock Knowledge Bases, all queries generated by the agent will execute flawlessly against the real RDS schema.
